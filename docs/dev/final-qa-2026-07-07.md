## Kill/restart scenarios

### SIGTERM

Initial finding: SIGTERM used the default process action and did not save totals/config before exit.

Regression test added to `tests/test_lifecycle.cpp`; it starts `build/nsl-linux --simulate --minimized`, sends SIGTERM, and asserts config exists with non-zero simulated totals.

Post-fix direct smoke:

```text
exit=0 config_exists=True
[totals]
monthKey=2026-07
rxMonth=81920
txMonth=81920
```

Fix: `main.cpp` now bridges SIGTERM/SIGINT through a pipe + `QSocketNotifier` into the Qt event loop; `MainWindow::shutdownForSignal()` saves totals/config before quitting.

### Crash / SIGKILL

Direct smoke after 65 seconds of `--simulate` then SIGKILL:

```text
exit=-9 config_exists=True
[totals]
monthKey=2026-07
rxMonth=778240
txMonth=737280
```

Conclusion: the 60-second totals flush bounds hard-crash loss to at most about 60 seconds. A crash before the first flush can lose that first partial minute.

## Wayland specifics

- Body drag path is implemented via `QWindow::startSystemMove()` from `MainWindow::mousePressEvent()` on left button press. This is the correct Wayland-safe path; synthetic non-pointer events cannot fully prove compositor drag behavior.
- Window geometry is fixed-size by design. The app persists window position and pane visibility; it does not persist arbitrary free-resize dimensions because the widget is intentionally fixed to the AnalogX-style layout.
- Always-on-top toggling was tightened: applying the setting now uses a live hide/show cycle so Qt Wayland window flags and LayerShellQt state are recreated without restarting the process. Expected user-visible limitation: a brief flicker when toggling.

## Residual Wayland limitations vs AnalogX on Windows

- Frameless Wayland windows depend on compositor support for `startSystemMove()`; behavior is KDE/Qt-specific rather than Win32 title-bar behavior.
- Layer-shell always-on-top is best-effort and KDE-specific when `layer-shell-qt` is available. Without it, users may still need the documented KWin “Keep above” rule.
- Tray behavior is StatusNotifierItem-based under Plasma, not a Windows notification-area icon. Synthetic context-menu activation may log Wayland popup-grab warnings because it is not a real pointer event.
- URL clipboard monitoring is constrained by Wayland security; Klipper DBus polling is the KDE fallback.
